## 验证深度标签

本文件的 `D0-D5` 与子代理复审提出的 `L0-L4` 口径等价：`D0/D1` 对应静态可实现性或数据/接口阻塞，`D2` 对应核心机制小实验，`D3` 对应 offline/frozen performance gate，`D3.5` 对应真实 batch 梯度 sanity，`D4-lite` 对应 1-2 epoch 短训趋势，`D4` 对应 seed42 training smoke，`D5` 对应多 seed 稳定性。后续文档统一使用这些标签，但必须同时写清 `claim_scope`。

| 标签 | 含义 | 能支持的结论 | 不能支持的结论 |
| --- | --- | --- | --- |
| `D0 Evidence-only` | 只做规划、源码分析、旧结果类比或 reserve audit | 只能作为风险先验 | 不能判方法 No-Go |
| `D1 Interface/Data-Blocked` | 核心输入、接口、teacher logits 或代码路径缺失 | 只能判 `Blocked` | 不能判方法失败 |
| `D2 Direct-Mechanism-Probe` | 直接让核心机制作用于目标张量/路径，观察 logit、feature、routing、flip、control moat | 可判当前机制响应是否成立；若方法本质是 inference/frozen，可判当前变体 No-Go | 不能判训练期机制一定无效 |
| `D3 Offline-Performance-Gate` | 用 train/val frozen features/logits 做 adapter、head、router、calibration 等离线性能 gate | 可判当前 offline/frozen 变体是否能提升结果 | 不能永久排除 end-to-end training 版本 |
| `D3.5 Gradient-Sanity` | 在真实训练 batch 上对候选 loss 做 forward/backward，记录 finite/nonzero 梯度、梯度范数、与 CE 梯度 cosine、触达 final classifier / `h_final` / branch path 的位置 | 可判当前 loss 公式是否不可导、无梯度、NaN、完全退化成 CE/置信度 control 或严重梯度冲突 | 不能判性能，不能替代训练 |
| `D4-lite Training-Dynamics-Smoke` | 1-2 epoch train/val-only mini-smoke，强 controls 同 budget，落盘训练曲线、val trend、flip/hard-bin | 可判是否值得冻结正式 D4 manifest | 不能判当前训练实现 No-Go，不能封 Primary |
| `D4 Seed42-Smoke` | seed42 train/val 小训，含强 baseline/control、flip audit、落盘 metrics | 可判当前训练实现是否值得复核 | 不能封最终主线 |
| `D5 Three-Seed-Val` | 三 seed val 复核且不用 test 选方法 | 可判稳定性，支持 `Primary-Candidate` 或强 No-Go | 仍不能用 test 回头改方法 |

判定规则：如果一个方法的核心机制是训练期改变 representation、loss、routing 或 final boundary，至少需要 `D4 Seed42-Smoke` 才能判当前训练实现 No-Go；如果只做了 `D2/D3`，只能写“当前 frozen/offline/proxy 变体 No-Go”。`D3.5/D4-lite` 是浅层与正式训练之间的补深度层，只能决定是否值得正式 D4，不能独立写训练实现 No-Go。如果一个方向只到 `D0/D1`，必须标为 `Provisional No-Go` 或 `Blocked` 并进入重验池。

## 结论作用域规则

每个候选必须显式记录：

- `claim_scope`：当前 probe、当前 offline/frozen 变体、当前训练实现，还是整个方法族。
- `level_reached`：已经达到的最高验证深度。
- `required_level_for_exclusion`：要排除该 claim_scope 至少需要的验证深度。
- `status_scope`：`No-Go`、`Blocked`、`Not-Unlocked`、`Provisional No-Go` 或 `Feasible-B` 的适用范围。

`Not-Unlocked` 是管理状态，不是实验失败。`Blocked` 是数据/接口前置没通，不是方法失败。`Provisional No-Go` 是旧证据或间接证据带来的风险先验，不是永久排除。任何 `[x]` 只能表示该项审计或记录完成，不能自动表示技术假设已充分验证。

## 三类复审归纳

本轮子代理日志审计后，所有历史候选按“是否真的跑到核心机制”归为三类：

1. **只分析未跑核心实验**：P1-B 真实 domain expert、R5-Prime / R5-A+D / R5-S reserve、R5-B/R5-C/R5-E/R5-P/R5-I、P2-E。这些不能写永久 No-Go，必须按上表补直接机制实验或接口/数据前置。R6-D/R6-G/R6-H/R6-J 已在 2026-05-28 补齐当前作用域直接实验。
2. **跑了但测试深度不够**：P1-A 的 soft/EMA prototype 延伸、P2-C 真训练 domain expert、P2-D/R3/R4/source-family 新 mixture、G0-A/G0-D/G0-S、R6-B 训练期聚合、R6-C 多 seed、R6-F consistency training。这些当前只否定已跑变体；若要排除训练期家族，必须新 manifest 并达到对应 D4/D5。
3. **实验足够不应重开当前实现**：selector/P1-C、旧 R3 regret router、P0-B late DCA/source-view 当前形态、P2-A/P2-B/P2-D、Round3 residual、Round4 offline boundary patches、G0-E/G0-S、R5-A 当前 smoke、R6-E early-exit inference diagnostic、R6-I 当前 frozen final-domain adapter。这些可以维持原判，但仍要写清 `status_scope`，避免误伤本质不同的新候选。

执行含义：防错杀不等于把所有旧路线拉回来重跑。优先资源只给“未触达核心机制但仍有第一性原理价值”的方向；已经被 D4/D5 或强 control moat 打穿的当前实现，不再空转。

## 执行底线

- 没有达到匹配验证深度，不写永久 No-Go。
- `D0/D1` 只能写 `Evidence-only`、`Provisional No-Go` 或 `Blocked`。
- `D2/D3` 只能否定当前 frozen/offline/proxy 变体，不能否定训练期家族。
- `D3.5/D4-lite` 只能提供梯度可达性和短训趋势证据，不能独立写训练实现 No-Go。
- 训练期方法必须先有独立 manifest、train/val-only、强 controls、落盘 metrics/flip audit，才能跑 `D4`。
- 三 seed val 过线前继续禁止 test 参与任何方法选择。
