# PANDA 创新方案赛马总控与实施协议

日期：2026-05-25

## 文档定位

本文件是 PANDA 创新阶段的总控协议。后续任何新方法都必须先进入这里定义的候选池，按同一套可行性验证、失败切换、可行标记、全量赛完、深入复核和最终主线确定流程执行。

本协议的目的不是多写方案，而是防止再次出现三类低效循环：

1. 只看某个单点 seed 或单个指标就升级为主方法。
2. 只改 selector / neighbor set，却没有证明 final classifier boundary 被改变。
3. 一个方案初步可行后立刻停止探索，导致错过更强、更稳、更能写成论文的方法。

当前结论必须按验证深度和作用域读取，不能把历史 `No-Go` 压缩成“方法族永久失败”。总览如下：

验证深度底线：`D0 Evidence-only` 与 `D1 Interface/Data-Blocked` 不能支持 No-Go，只能支持 Evidence-only、`Blocked` 或 `Provisional No-Go`；`D2 Direct-Mechanism-Probe` 与 `D3 Offline-Performance-Gate` 只能否定当前 frozen/offline/proxy 变体；`D3.5 Gradient-Sanity` 只验证真实 batch 梯度是否 finite/nonzero、是否触达目标路径、是否退化成 CE/置信度 control；`D4-lite Training-Dynamics-Smoke` 只看 1-2 epoch 同预算短训趋势；若方法核心是训练期 loss、representation、routing 或 final-boundary 改造，至少需要 `D4 Seed42-Smoke` 才能判当前训练实现 No-Go；只有 `D5 Three-Seed-Val` 才能支持稳定性主张。测试深度按方法而定，目标是验证核心机制是否真的能提升 train/val 结果，而不是机械要求所有候选都跑同一种最小测试。

状态作用域底线：每个候选必须记录 `claim_scope`、`level_reached`、`required_level_for_exclusion`、`status_scope`。`Not-Unlocked` 只是管理状态，`Blocked` 只是接口/数据前置没通，`Provisional No-Go` 只是旧证据风险先验；这三者都不能被解读为方法族失败。已 D4/D5 或强 control moat 失败的当前实现不重开，但 frozen/proxy/static audit 不能永久排除本质不同的训练期机制。

## 总原则

### 1. 先验来自第一性原理，不来自调参惯性

PANDA 的核心链路是：

```text
frozen text/image/CLIP features
 -> text/image/fusion experts
 -> h_di
 -> domain prototypes / PAD
 -> source-domain selector
 -> DCA
 -> h_collab
 -> final_classifier_panda([h_di, h_collab])
```

已有失败规律说明：

- reliability / uncertainty 信号有诊断价值，但只重排 PAD/Gumbel logits 不稳定。
- 大幅改变 neighbor set 不等于改变 final label。
- R3 soft router 会塌缩到 self domain，non-self source alpha 接近 0。
- seed42 小幅过线不可靠，seed2024 已多次暴露方法脆弱性。

因此新方案必须回答一个更硬的问题：

```text
这个模块是否真正改变了 final classifier boundary，
并且该改变不能被 deterministic、self、duplicate、random、bottom、shuffled controls 解释？
```

### 2. 先验可行性 gate 早于训练

每个候选方法都要先做与核心机制匹配的可行性验证。若核心机制在 frozen checkpoint 或 batch smoke 中已经不可实现，也必须按 `level_reached` 落状态：`D0/D1` 不得写 No-Go，`D2/D3` 只关闭当前 probe/offline/frozen/proxy 变体；只有当 `required_level_for_exclusion` 已满足，才允许不进入下一层训练。

例子：

- R4 必须先证明 forced non-self source views 对 final logits 有非噪声扰动。
- Feature-aware PAD 必须先证明 PAD top2 non-self view 与 random/bottom2 有差异。
- Domain-conditioned expert gate 必须先证明专家门控不是简单修复 domain 0 gate，且能影响 final logits。
- Evidential/reliability extension 必须先证明它不是只改善校准、同时损害 F1/Acc。

### 3. 可行不等于胜出

如果某个方案先通过可行性 gate，只标记为可行，不立刻封主线。必须继续验证候选池里其他 P0/P1 方案的最小可行性。所有候选都完成后，再对可行方案做深入复核。

新增执行状态机：

```text
Pending -> Running-Gate0 -> No-Go / Feasible-A / Feasible-B
-> All-Candidates-Gated -> Deep-Validation -> Primary-Candidate -> Mainline
```

强制规则：

- `No-Go`：只在当前证据深度达到该 `claim_scope` 的 `required_level_for_exclusion` 时使用。核心逻辑失败、强对照打穿、需要 test correctness 才能成立、或不能影响 final boundary，都必须先判断是 `D2/D3` 当前变体失败，还是 `D4` 当前训练实现失败。
- `Feasible-A/B`：只表示核心机制暂时成立；必须继续验证候选池内所有其他方案，禁止提前封主线。
- `All-Candidates-Gated`：同一轮全部活跃候选都有 `No-Go/Feasible-A/Feasible-B` 标签后，才允许进入深入复核。
- `Primary-Candidate`：只能从深入复核后的可行候选中产生；单个 Gate-0、单个 seed42 smoke 或单个指标过线都不能越级。
- 若深入复核阶段发现可行候选被更强 control 打穿，降级为 `No-Go` 或 `Feasible-B`，再从剩余可行候选中继续排序；若没有剩余可行候选，则开启下一轮第一性原理候选池。

状态标签：

| 标签 | 含义 | 下一步 |
| --- | --- | --- |
| `Pending` | 尚未验证 | 排队做最小 gate |
| `No-Go` | 当前 `claim_scope` 的核心机制不可行或强对照打穿，且 `required_level_for_exclusion` 已满足 | 停止该 claim_scope，不外推到方法族 |
| `Feasible-A` | 机制成立且强对照未打穿 | 继续验证其他候选，暂不封主线 |
| `Feasible-B` | 机制成立但论文张力或指标不足 | 作为消融/附录/二阶段 |
| `Primary-Candidate` | 所有候选赛完后排名第一 | 进入三 seed val 复核 |
| `Mainline` | 三 seed val 过线且 test 仍锁定为 confirmatory | 才能写正式方法主表 |

### 4. Test 永远不是方案选择器

三 seed val 没过线前，不导出、不打开、不分析 test。不能根据 test 改 `lambda`、view source、reliability weight、弱域定义、primary config 或论文主张。

## 外部文献借鉴原则

只借鉴能解释当前失败规律的外部方法，不为了堆相关工作而加模块。

当前可借鉴点：

- PANDA：我们的增量必须站在 `DMPG + PAD + GNS + DCA` 之后，不重复“选择相似邻域”。
- DomainBed：严格 model selection、多 seed、强对照，比单点指标更重要。
- GroupDRO / REx：worst-group / risk variance 可以做二阶段 robustness，但不能在主机制未成立前加进 primary。
- MoE / Switch Transformer：load balance 和 expert collapse 诊断可借鉴；但 PANDA 的 domain gate 修复不能包装成大模型 MoE 创新。
- Evidential learning：适合做可信性扩展，不适合在 R4 前抢主线。
- MIMoE-FND 等多模态 MoE：可借鉴 modality interaction/gating 的写法，但当前核心问题是 source-domain utility，不是普通模态融合。
